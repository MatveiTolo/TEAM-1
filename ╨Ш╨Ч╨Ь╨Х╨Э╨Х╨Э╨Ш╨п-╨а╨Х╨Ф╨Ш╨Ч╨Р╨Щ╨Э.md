# CAESAR — редизайн интерфейса и доработка эндпоинтов

Краткий отчёт обо всех изменениях. Страницы **Landing / Login / Register оставлены без изменений** (как и требовалось) — они уже выполнены в тёмном имперском стиле и служат эталоном дизайна.

---

## 1. Единая дизайн-система (новое)

- `caesar.client/src/styles/theme.css` — **новый источник правды** для всех внутренних страниц: CSS-переменные (фон, стеклянные поверхности, границы, текст, семантика, цвета статусов доски, радиусы, шрифты Cinzel/Inter, тени, высота топбара) + переиспользуемые классы (`.caesar-bg`, `.glass-card`, `.glass-panel`, `.caesar-input`, `.caesar-btn--*`, `.caesar-title`, `.caesar-alert`, `.caesar-badge`, `.caesar-loader` и т.д.). Подключается в `main.tsx` первым.
- `caesar.client/src/components/AppShell/` — **новая обёртка** для внутренних экранов: фиксированный стеклянный топбар (логотип, заголовок, навигация с иконками, активный пункт, кнопка «Выйти»), анимированный фон и контейнер контента. Заменила 6 продублированных блоков топбара/бейджа в `App.tsx`.

## 2. Редизайн внутренних страниц (тёмная тема)

Переведены в тёмный глассморфизм: **Projects, ProjectSetup, UserProfile, Admin, Invite, Error, Board / Column / TaskCard**, а также **обе модалки доски** — `CreateTaskModal` и `TaskDetailsModal` (полностью переписан CSS; селектор `.form-group` в модалке заскоплен под `.modal`, чтобы не конфликтовать с глобальным `.form-group` из ProjectSetup).

`App.tsx` переписан (379 → ~169 строк): убрано дублирование, внутренние экраны обёрнуты в `AppShell`.

## 3. Новые рабочие страницы (раньше были заглушки)

- **Calendar** — настоящая сетка месяца (неделя с понедельника), задачи раскладываются по дедлайнам, чипы окрашены по статусу, навигация назад/вперёд/сегодня, легенда.
- **Reports** — дашборд статистики из задач: всего/выполнено %/скоро дедлайн/просрочено, прогресс-бары по статусам, список ближайших дедлайнов.

## 4. Фронтенд: логика и исправления

- **Исправлен критический баг drag-and-drop** (`hooks/useTasks.ts`): теперь на `/api/Tasks/{id}/move` уходит правильный DTO `{ targetTaskStatus: <1..4>, newPosition }` вместо `{ status }`. Добавлена карта статусов строка↔число.
- **`createTask`** теперь отправляет корректный `CreateTaskDto` `{ projectPageId, title, description, deadline }`.
- Добавлен **`normalizeTask()`** — преобразование ответа бэкенда (camelCase, статус числом) во фронтовый тип `Task` (snake_case). Применяется в загрузке/создании/перемещении задач.
- **`services/mockApi.ts`** приведён к реальному API с мок-фолбэком (UI остаётся демонстрируемым без бэкенда):
  - `updateTask` — корректный `PATCH /api/Tasks/{id}` (поле `assignedToId`);
  - `getTaskHistory`, `getComments` — нормализация под формат модалки; добавлен `addComment`;
  - `getUsers` → `GET /api/Users`, `getProjects` → `GET /api/Projects` (с форматированием даты и подписью роли), `getUserProfile` → `GET /api/Users/me`, `createProject` → `POST /api/Projects`, `updateUser` → `PATCH /api/Users/{id}`.

## 5. Бэкенд: реализация и исправления эндпоинтов

- **`ProjectPagesController`** — добавлен `[Authorize]`, убран хардкод `currentUserId = 1`; пользователь берётся из JWT.
- **`TasksController`** — реализован **`PATCH /api/Tasks/{id}`** (изменение названия/описания/дедлайна/исполнителя) с проверкой прав участника и записью в историю.
- **`UsersController`** — добавлены **`GET /api/Users`** (список для админа), **`GET /api/Users/me`** (свой профиль), **`PUT /api/Users/profile`** (обновление своего профиля из JWT).
- **`ProjectsController` + `Project` + `CreateProjectDto`** — добавлено поле **`Theme`** (ранее тема игнорировалась); `GET /api/Projects` теперь возвращает `Theme`, `CreatedAt`, `TasksCount`.
- **`AuthController`** — регистрация теперь возвращает `{ Token, Username }` (авто-вход), генерация токена вынесена в общий метод `GenerateToken`.
- **`Program.cs`** — добавлен CORS-полиси для дев-окружения; `UseHttpsRedirection()` теперь только вне Development (чтобы не ломать Vite-прокси на `http://localhost:5254`).
- **Новый DTO** `UpdateTaskDto`.
- **Миграция БД** `20260627000000_AddThemeToProject` (+ Designer + обновлён snapshot) — добавляет колонку `Theme` в `Projects`. Применится автоматически при старте (`MigrateAsync`).

## 6. Оптимизация и чистка

- Удалены мёртвые файлы: `api/apiClient.ts`, `api/mockData.ts`, `App.css` (не импортировался), `RevealCard.tsx` (не использовался и тянул отсутствующий пакет `motion/react`).
- `index.css` переписан со светлой «мраморной» базы на тёмную (страницы Login/Register/Landing задают свой фон и не затрагиваются).
- Убран шумный `console.log` из внутренних файлов (ошибки `console.error` сохранены).
- Сборка фронтенда проходит чисто: **`tsc -b` без ошибок типов**, `vite build` успешен.

## Что нужно проверить у себя (вне песочницы)

- **Запуск с реальной БД PostgreSQL**: миграция `Theme` применится автоматически; стоит прогнать создание проекта/задачи и drag-and-drop на живом бэкенде.
- **`dotnet build`**: C# писался без локального компилятора (в песочнице нет dotnet SDK) — рекомендуется собрать решение и прогнать у себя.
- **Vite dev-сертификат**: `vite.config.ts` сделан устойчивым — если `dotnet dev-certs` недоступен, dev-сервер не падает, а автоматически откатывается на HTTP (на вашей машине с .NET SDK HTTPS-сертификат выпустится как обычно, поведение не меняется).
- **Отсутствующий ассет `BenzFon.jpg`**: на странице Landing (`GlassCard3D.css`) есть ссылка на `/attachments/BenzFon.jpg`, но самого файла нет в репозитории. Это **существующая проблема** (страница в защищённом наборе Landing/Login/Register, её дизайн не трогался). Положите файл в `caesar.client/public/attachments/`, иначе фон карточки на лендинге не загрузится.
- **Telegram-уведомления**: архитектура `NotificationService`/`Factory` присутствует, но провайдеры в `Program.cs` намеренно не зарегистрированы — **вне рамок этой задачи** (нужно подключить и зарегистрировать провайдер отдельно).
- **Заметка**: архив включает папку `.git`, так что вы сможете закоммитить и запушить изменения со своей машины (отсюда пуш недоступен).
